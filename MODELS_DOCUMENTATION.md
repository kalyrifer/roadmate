# Техническая документация моделей Users и Trips

## Модель Users (Пользователь)

### Назначение таблицы

Таблица `users` — центральная сущность системы, представляющая участников сервиса совместных поездок. Один аккаунт может выступать как водитель (создаёт поездки) и как пассажир (бронирует поездки). Это универсальная модель, объединяющая обе роли в одном пользователе.

### Типы данных и ограничения

**email**
- Тип: VARCHAR(255)
- Ограничения: NOT NULL, UNIQUE, INDEX
- Назначение: основной идентификатор для входа, используется в аутентификации

**password_hash**
- Тип: VARCHAR(255)
- Ограничения: NOT NULL
- Назначение: хэш пароля, хранитсяbcrypt-формате, пароль никогда не сохраняется в открытом виде

**rating_average**
- Тип: NUMERIC(3,2) — хранит значения от 0.00 до 5.00
- Ограничения: NOT NULL, DEFAULT 0.00, INDEX
- Назначение: материализованный рейтинг для быстрого отображения без вычислений

**role**
- Тип: ENUM (UserRole)
- Значения: `user` (обычный пользователь), `admin` (администратор)
- Ограничения: NOT NULL, DEFAULT user, INDEX

**language**
- Тип: VARCHAR(10)
- Ограничения: NOT NULL, DEFAULT 'ru'
- Поддерживаемые значения: 'ru', 'en'

**deleted_at**
- Тип: TIMESTAMP (nullable)
- Назначение: soft delete — при удалении пользователя данные не удаляются физически, а помечаются временной меткой

### Статусы и enum-поля

**UserRole**
- `user` — обычный пользователь
- `admin` — администратор системы

**Флаги статуса**
- `is_active`: TRUE — аккаунт активен, FALSE — деактивирован
- `is_blocked`: TRUE — заблокирован администратором

### Рекомендации по расширению

1. **Верификация**
   - Поле `verification_token` — токен подтверждения email
   - Поле `password_reset_token` — токен сброса пароля
   - Поле `password_reset_expires` — срок действия токена

2. **Внешние интеграции**
   - Поле `telegram_chat_id` — для уведомлений в Telegram
   - Поле `fcm_token` — для push-уведомлений

3. **Водительские данные**

   - Поле `driver_license_verified` — верификация прав

4. **Социальные данные**
   - Поле `social_links` (JSONB) — ссылки на соцсети

## Модель Trips (Поездка)

### Назначение таблицы

Таблица `trips` — основная бизнес-сущность системы, представляющая поездку, которую создаёт водитель. Поездка характеризуется маршрутом, датой и временем, ценой за место, вместимостью и параметрами (багаж, курение, животные и т. д.).

### Поля модели Trips

| Поле | Тип | Обязательное | Индекс | Описание |
|------|-----|--------------|--------|----------|
| `id` | UUID | Да | Да (PK) | Уникальный идентификатор |
| `driver_id` | UUID | Да | Да | ID водителя (FK на users) |
| `from_city` | VARCHAR(100) | Да | Да | Город отправления |
| `from_address` | VARCHAR(255) | Нет | Нет | Точный адрес отправления |
| `to_city` | VARCHAR(100) | Да | Да | Город назначения |
| `to_address` | VARCHAR(255) | Нет | Нет | Точный адрес назначения |
| `departure_date` | DATE | Да | Да | Дата поездки |
| `departure_time_start` | TIME | Да | Нет | Время отправления (начало) |
| `departure_time_end` | TIME | Нет | Нет | Время отправления (конец) |
| `is_time_range` | BOOLEAN | Да | Нет | Используется ли диапазон |
| `arrival_time` | TIME | Нет | Нет | Ориентировочное время прибытия |
| `price_per_seat` | NUMERIC(10,2) | Да | Да | Цена за место |
| `total_seats` | INTEGER | Да | Нет | Всего мест |
| `available_seats` | INTEGER | Да | Да | Доступные места |

| `music_allowed` | BOOLEAN | Да | Нет | Музыка разрешена |
| `pets_allowed` | BOOLEAN | Да | Нет | Животные разрешены |
| `car_model` | VARCHAR(100) | Нет | Нет | Модель автомобиля |
| `car_color` | VARCHAR(50) | Нет | Нет | Цвет автомобиля |
| `car_license_plate` | VARCHAR(20) | Нет | Нет | Номерной знак |
| `status` | ENUM | Да | Да | Статус поездки |
| `cancelled_at` | TIMESTAMP | Нет | Нет | Время отмены |
| `cancelled_reason` | TEXT | Нет | Нет | Причина отмены |
| `created_at` | TIMESTAMP | Да | Нет | Время создания |
| `updated_at` | TIMESTAMP | Да | Нет | Время обновления |
| `deleted_at` | TIMESTAMP | Нет | Да | Мягкое удаление |

### Типы данных и ограничения

**driver_id**
- Тип: UUID
- Ограничения: NOT NULL, FK на users.id, INDEX
- ON DELETE: SET NULL — при удалении водителя поездка сохраняется, но становится "осиротевшей"

**price_per_seat**
- Тип: NUMERIC(10,2)
- Ограничения: NOT NULL, INDEX
- Назначение: точность до 2 знаков после запятой

**total_seats / available_seats**
- Тип: INTEGER
- Ограничения: NOT NULL, available_seats индексируется
- Бизнес-логика: available_seats не может быть больше total_seats, проверяется на уровне приложения

**departure_time_start / departure_time_end**
- Тип: TIME (с timezone)
- Логика: если is_time_range = FALSE, то используется только departure_time_start; если TRUE — используется диапазон

**is_time_range**
- Тип: BOOLEAN
- Ограничения: NOT NULL, DEFAULT FALSE
- Назначение: определяет, как отображать время — точно или диапазоном

**status**
- Тип: ENUM (TripStatus)
- Ограничения: NOT NULL, DEFAULT draft, INDEX

| Индекс | Поле | Назначение |
|--------|------|------------|
| `ix_trips_driver_id` | driver_id | Поиск поездок водителя |
| `ix_trips_from_city` | from_city | Фильтрация по городу отправления |
| `ix_trips_to_city` | to_city | Фильтрация по городу назначения |

| `ix_trips_deleted_at` | deleted_at | Исключение удалённых |
| `ix_trips_route_search` | from_city, to_city, departure_date, status | Композитный индекс для главного поиска |

### Внешние ключи и связи

Связь | Таблица | Тип | ON DELETE | Описание
------|---------|-----|-----------|----------
driver | users | N:1 | SET NULL | Водитель поездки
trip_requests | trip_requests | 1:N | CASCADE | Заявки на бронирование
conversations | conversations | 1:N | SET NULL | Связанные чаты
reviews | reviews | 1:N | SET NULL | Отзывы о поездке

### Статусы и enum-поля

**TripStatus**
- `draft` — черновик, поездка не опубликована
- `published` — опубликована, доступна для бронирования
- `active` — активна (в процессе выполнения)
- `completed` — завершена
- `cancelled` — отменена

### Рекомендации по расширению

1. **Маршрут**
   - Поле `intermediate_stops` (JSONB) — промежуточные точки
   - Поле `route_coordinates` (JSONB) — координаты для карты
   - Поле `estimated_distance` — расстояние в км
   - Поле `estimated_duration` — время в пути

2. **Повторяющиеся поездки**
   - Поле `recurrence_type` — тип повторения (ежедневно, еженедельно)
   - Поле `recurrence_end_date` — дата окончания повторений

3. **Оплата**
   - Поле `payment_method` — способ оплаты
   - Поле `is_prepaid` — предоплата

4. **Автоматизация**
   - Поле `auto_cancel_after` — автоотмена при отсутствии брони

## Как эти таблицы станут основой для других сущностей

### trip_requests (Заявки на бронирование)

Связана с trips через foreign key `trip_id`. Одна поездка может иметь множество заявок от разных пассажиров. Заявка ссылается на пользователя (`passenger_id`) как на создателя.

Связь: Trip → TripRequest (1:N)

### conversations (Диалоги)

Диалоги привязываются к поездке или заявке. Чат создаётся после подачи заявки или при первом сообщении между пользователями в контексте поездки.

Связи: Trip → Conversation (1:N), User → Conversation (N:1)

### messages (Сообщения)

Сообщения принадлежат диалогу, который в свою очередь связан с поездкой. Отправитель — всегда пользователь.

Связи: Conversation → Message (1:N), User → Message (N:1)

### reviews (Отзывы)

Отзывы создаются после завершения поездки. Связаны с поездкой и двумя пользователями — автором отзыва и тем, кому он адресован.

Связи: Trip → Review (1:N), User → Review как автор (N:1), User → Review как получатель (N:1)

### notifications (Уведомления)

Генерируются при событиях, связанных с поездками, заявками, сообщениями, отзывами. Все уведомления привязаны к пользователю-получателю.

Связь: User → Notification (1:N), опционально Trip/TripRequest/Conversation

### user_settings (Настройки пользователя)

Связь ONE-TO-ONE с users. Создаётся автоматически при регистрации пользователя. Удаляется каскадом при удалении пользователя.

Связь: User → UserSettings (1:1)

### audit_logs (Логи действий)

Записывают все действия пользователей в системе: создание поездки, подача заявки, отправка сообщения и т. д.

Связь: User → AuditLog (1:N)

## Почему такая структура подходит для сервиса совместных поездок

1. **Универсальность пользователя** — один аккаунт работает и как водитель, и как пассажир, без дублирования данных в отдельных таблицах

2. **Оптимизация поиска поездок** — композитный индекс на (from_city, to_city, departure_date, status) обеспечивает быстрый поиск по главному сценарию использования

3. **Гибкость времени** — поддержка точного времени и диапазона через is_time_range позволяет водителю указать удобное окно отправки

4. **Контроль вместимости** — отдельные поля total_seats и available_seats с проверкой на уровне приложения предотвращают перепродажу мест

5. **Аудируемость** — soft delete через deleted_at сохраняет историю для аналитики и расследований, но не занимает место в основных запросах

6. **Материализованный рейтинг** — хранение rating_average в таблице users избавляет от необходимости вычислять среднее значение при каждом запросе

7. **Расширяемость** — JSONB-поля и enum-типы позволяют добавлять новую функциональность без изменения структуры таблиц
